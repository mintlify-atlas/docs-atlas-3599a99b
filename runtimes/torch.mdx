pyptx detects `torch.Tensor` inputs automatically. When you call a `@kernel` with PyTorch tensors, the dispatch routes to the Torch runtime path instead of the JAX FFI path — no extra wrappers needed.

## Installation

<Steps>
  <Step title="Install PyTorch extras">
    Install the Torch extras group, which requires `torch>=2.1`.

    ```bash
    pip install 'pyptx[torch]'
    ```
  </Step>
  <Step title="Install ninja for the fast C++ dispatch path (optional)">
    When `ninja` is available, pyptx JIT-compiles a C++ pybind11 extension (`pyptx/_shim/torch_ext.cpp`) on first use and caches it. This cuts per-call overhead from ~40 µs (ctypes) to ~10 µs.

    ```bash
    pip install ninja
    ```

    <Tip>
      The extension is compiled once and cached on disk. Subsequent process restarts skip compilation entirely.
    </Tip>
  </Step>
</Steps>

## Complete example

The kernel below is identical to the JAX example. Passing `torch.Tensor` inputs at call time is all that changes.

```python
import torch
from pyptx import Tile, kernel, ptx, reg
from pyptx.types import f32, u32

WARP_SIZE = 32


def build_rms_norm(B: int, N: int, eps: float = 1e-6):
    items_per_thread = N // WARP_SIZE

    @kernel(
        in_specs=(Tile(B, N, f32), Tile(N, f32)),
        out_specs=(Tile(B, N, f32),),
        grid=(B, 1, 1),
        block=(WARP_SIZE, 1, 1),
        arch="sm_90a",
    )
    def rms_norm(X, W, Y):
        px, pw, py = ptx.global_ptrs(X, W, Y)
        row = reg.from_(ptx.special.ctaid.x(), u32)
        row_byte_off = row * (N * 4)
        px += row_byte_off
        py += row_byte_off

        tid = reg.from_(ptx.special.tid.x(), u32)
        sum_sq = reg.scalar(f32, init=0.0)
        x_vals = reg.array(f32, items_per_thread)

        for i in range(items_per_thread):
            idx = tid + i * WARP_SIZE
            ptr = px + idx * 4
            ptx.inst.ld.global_.f32(x_vals[i], ptx.addr(ptr))
            ptx.inst.fma.rn.f32(sum_sq, x_vals[i], x_vals[i], sum_sq)

        ptx.warp.reduce_sum(sum_sq)
        mean_sq = reg.scalar(f32)
        inv_n = reg.scalar(f32, init=1.0 / N)
        eps_reg = reg.scalar(f32, init=eps)
        rstd = reg.scalar(f32)
        ptx.inst.mul.f32(mean_sq, sum_sq, inv_n)
        ptx.inst.add.f32(mean_sq, mean_sq, eps_reg)
        ptx.inst.rsqrt.approx.f32(rstd, mean_sq)

        for i in range(items_per_thread):
            idx = tid + i * WARP_SIZE
            off = idx * 4
            w_val = reg.scalar(f32)
            y_val = reg.scalar(f32)
            ptx.inst.ld.global_.f32(w_val, ptx.addr(pw + off))
            ptx.inst.mul.f32(y_val, x_vals[i], rstd)
            ptx.inst.mul.f32(y_val, y_val, w_val)
            ptx.inst.st.global_.f32(ptx.addr(py + off), y_val)
        ptx.ret()

    return rms_norm


k = build_rms_norm(16, 512)

x = torch.randn(16, 512, device="cuda") * 0.3
w = torch.randn(512, device="cuda") * 0.1 + 1.0
out = k(x, w)  # dispatches to Torch path automatically
```

Output tensors are allocated by pyptx from `out_specs` on the same device as the first input. The returned tensor is a standard `torch.Tensor` with no special properties.

## Dispatch tiers

pyptx provides three dispatch tiers, ordered from lowest to highest overhead.

### CUDA graph replay — ~4 µs

Capturing a pyptx kernel into a `torch.cuda.CUDAGraph` eliminates almost all CPU overhead. This is the fastest path and works with every pyptx kernel.

```python
k = build_rms_norm(256, 4096)
x = torch.randn(256, 4096, device="cuda")
w = torch.randn(4096, device="cuda")

# Warm up: first call compiles PTX and registers the kernel
k(x, w)

# Capture
g = torch.cuda.CUDAGraph()
s = torch.cuda.Stream()
s.wait_stream(torch.cuda.current_stream())
with torch.cuda.stream(s):
    k(x, w)
torch.cuda.current_stream().wait_stream(s)
with torch.cuda.graph(g, stream=s):
    out = k(x, w)

# Replay — ~4 µs per call
g.replay()
```

<Warning>
  CUDA graph replay fixes input and output memory addresses. If you update `x` or `w` in-place between replays the graph will use the new data. Allocating new tensors between replays is not allowed while the graph is active.
</Warning>

### Turbo eager — ~15–20 µs

On repeated calls with the same input shapes, pyptx caches the full launch configuration in `_turbo_torch` on the `Kernel` instance and bypasses all Python dispatch overhead on the hot path. This happens automatically with no user action required.

```python
k = build_rms_norm(256, 4096)
x = torch.randn(256, 4096, device="cuda")
w = torch.randn(4096, device="cuda")

# First call: full dispatch including PTX compilation (~40 µs)
out = k(x, w)

# Subsequent calls: turbo cache hit (~15–20 µs)
out = k(x, w)
```

The turbo path is only engaged when `ninja` is installed and the C++ extension has been compiled. It falls back to ctypes otherwise (~40 µs).

### Ctypes eager — ~40 µs

When `ninja` is not installed, the fallback path calls `pyptx_shim_launch_raw` via ctypes. It is correct and complete but slower due to Python ↔ C boundary marshaling.

## The `torch.library.custom_op` dispatch path

The Torch runtime path routes all calls through `torch.library.custom_op`. Each `(kernel, cubin_handle)` pair gets a unique op name (`pyptx::kernel_<handle>`) registered once in `get_or_register_torch_op`. This registration has two implementations:

- **Real implementation** — calls `call_kernel_via_torch`, which allocates output tensors, builds a `void**` pointer array, and calls `pyptx_shim_launch_raw` on the current CUDA stream.
- **Fake/meta implementation** — registered via `pyptx_op.register_fake`. During `torch.compile` tracing it allocates empty tensors with the right shape and dtype without touching the GPU, allowing Dynamo to infer output shapes and keep the kernel in the graph.

This is what makes the same kernel object work in both eager mode and `torch.compile` without any user-visible difference.

## The C++ extension shim

When `ninja` is available, `torch_support.py` calls `torch.utils.cpp_extension.load` on `pyptx/_shim/torch_ext.cpp` at first use. This pybind11 module exposes a single `launch_kernel` function that:

1. Accepts a handle, stream pointer, input tensors, output shapes, and output dtypes.
2. Allocates output tensors directly in C++.
3. Calls `pyptx_shim_launch_raw` without going through Python ctypes.

The result is roughly 3× less overhead than the ctypes path.

## Autograd with `differentiable_kernel`

pyptx supports `torch.autograd` through `differentiable_kernel`. You supply a forward kernel and a backward kernel; pyptx wraps them in a `torch.autograd.Function` subclass.

```python
from pyptx import differentiable_kernel

fwd = build_my_forward(M, N)
bwd = build_my_backward(M, N)

my_op = differentiable_kernel(
    fwd, bwd,
    save_for_backward=[0, 1],  # save input tensors at indices 0 and 1
)

x = torch.randn(M, N, device="cuda", requires_grad=True)
w = torch.randn(N, device="cuda", requires_grad=True)
out = my_op(x, w)
out.sum().backward()  # dispatches to bwd kernel with saved tensors + grad_output
print(x.grad, w.grad)
```

The backward kernel receives `(*saved_tensors, *grad_outputs)` as positional arguments and must return one gradient tensor per differentiable input. Any inputs beyond `num_grad_inputs` receive `None` gradients automatically.

<Note>
  `differentiable_kernel` uses `torch.no_grad()` inside both `forward` and `backward` to prevent accidental second-order differentiation through the PTX launch.
</Note>

## Constraints

- Inputs must be CUDA tensors. CPU tensors raise immediately.
- Inputs are expected to be contiguous. Non-contiguous tensors may produce incorrect results without raising an error.
- Each distinct input shape set results in a separate compiled kernel. Shape changes trigger a PTX retrace and `cuModuleLoadData` call.
