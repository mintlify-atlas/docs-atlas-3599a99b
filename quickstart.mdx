This page walks you through the complete pyptx workflow: define a kernel, inspect the PTX it produces, then launch it from PyTorch or JAX. The whole thing takes under five minutes. If you haven't installed pyptx yet, start with [installation](/installation).

<Steps>
  <Step title="Install pyptx">
    Install the base package plus whichever runtime you want to use for launching:

    <CodeGroup>

    ```bash PyTorch
    pip install ninja 'pyptx[torch]'
    ```

    ```bash JAX
    pip install 'pyptx[jax]'
    ```

    ```bash Both
    pip install ninja 'pyptx[all]'
    ```

    </CodeGroup>

    The `ninja` package is optional but recommended for the PyTorch path — it cuts dispatch overhead from ~34 µs to ~14 µs on first launch.
  </Step>

  <Step title="Define a kernel">
    A `@kernel`-decorated function runs at trace time on the host CPU, emitting PTX instructions as it goes. Every `ptx.*` call records exactly one PTX instruction.

    ```python
    from pyptx import kernel, reg, ptx
    from pyptx.types import f32, u32

    @kernel(arch="sm_90a")
    def tiny():
        tid = reg.scalar(u32)
        ptx.inst.mov.u32(tid, ptx.special.tid.x())
        x = reg.scalar(f32, init=0.0)
        ptx.inst.cvt.rn.f32.u32(x, tid)
        ptx.ret()
    ```

    Key points:

    - `arch="sm_90a"` targets Hopper; use `"sm_100a"` for Blackwell
    - `reg.scalar(u32)` allocates a PTX register — it runs in Python, not on the GPU
    - `ptx.inst.mov.u32(tid, ptx.special.tid.x())` emits a single `mov.u32` instruction
    - `reg.scalar(f32, init=0.0)` encodes the Python float to PTX's `0f00000000` hex form automatically
    - `ptx.ret()` closes the kernel body
  </Step>

  <Step title="Inspect the PTX">
    Call `.ptx()` on the kernel object before running anything. This is the actual PTX that will be handed to the driver — no hidden transformations.

    ```python
    print(tiny.ptx())
    ```

    Output:

    ```ptx
    .version 8.0
    .target sm_90a
    .address_size 64

    .visible .entry tiny(
    )
    {
        .reg .u32 %r<2>;
        .reg .f32 %f<1>;

        mov.u32 %r0, %tid.x;
        cvt.rn.f32.u32 %f0, %r0;
        ret;
    }
    ```

    What you write in Python maps directly to what appears here. There is no register renaming, no instruction reordering, and no fusion.
  </Step>

  <Step title="Launch from PyTorch">
    With `pyptx[torch]` installed, the kernel object works as a standard callable. Pass CUDA tensors directly:

    ```python
    from examples.hopper.rms_norm import build_rms_norm
    import torch

    k = build_rms_norm(4, 64)

    x = torch.randn(4, 64, device="cuda") * 0.3
    w = torch.randn(64, device="cuda") * 0.1 + 1.0
    out = k(x, w)
    ```

    You can also pass it to `torch.compile` without changing the kernel at all:

    ```python
    @torch.compile
    def compiled_rms(x, w):
        return k(x, w)

    out = compiled_rms(x, w)
    ```

    <Tip>
      On the first call, pyptx JIT-compiles the PTX through `cuModuleLoadData` and (if `ninja` is installed) builds the C++ dispatch extension. Subsequent calls reuse the compiled module from the cache.
    </Tip>
  </Step>

  <Step title="Launch from JAX">
    With `pyptx[jax]` installed, the kernel registers as a typed FFI handler and runs inside `jax.jit` with no code changes:

    ```python
    import jax
    import jax.numpy as jnp
    from examples.hopper.rms_norm import build_rms_norm

    k = build_rms_norm(4, 64)

    x = jnp.array([[...]], dtype=jnp.float32)  # shape (4, 64)
    w = jnp.ones(64, dtype=jnp.float32)

    @jax.jit
    def run(x, w):
        return k(x, w)

    out = run(x, w)
    ```

    The kernel object is the same object you would call from PyTorch. pyptx dispatches to the right runtime path automatically based on which framework's arrays are passed in.
  </Step>
</Steps>

## Using the transpiler

If you have existing PTX — from `nvcc`, Triton, Pallas, or cuBLAS — you can convert it to editable pyptx Python in one command:

```bash
python -m pyptx.codegen kernel.ptx --sugar --name my_kernel > my_kernel.py
```

The `--sugar` flag collapses spin-loops into `ptx.loop(...)`, demangles register names, and groups expression chains. The round-trip is byte-identical on 218+ corpus files.
